# 08 - Demo Guide

How to present Reporium to different audiences.

---

## What to Show First

Start with the live API. Open `https://reporium-api-573778300586.us-central1.run.app/docs` in a browser. This is the interactive Swagger UI.

Why start here:
- It is live, not a slide deck
- It responds in real time
- It proves the system works, not just that it was designed

Hit `/stats` first. Show the numbers: 826 repos, 29 languages. These are live from Neon, not hardcoded.

Then hit `/search?q=python`. Show that search works across all 826 repos.

## Nightly Pipeline Demo

Walk through what happens at 02:00 UTC every night:

1. Open the forksync repo. Show SYNC_REPORT: 143s, 792 checked, 1 error.
2. Open LAST_RUN: 9 API calls, 4,876 rate limit remaining.
3. Open reporium-db. Show index.json: 826 entries.
4. Open a per-repo JSON file. Show the metadata structure.
5. Hit the API /stats endpoint again. Show the numbers match.

The key message: every number in the API traces back to a file that was generated by the pipeline. Nothing is fabricated.

## forksync v1 to v2 Story

This is the strongest technical narrative. Tell it as a story:

**v1:** "We started with the simplest thing that could work. Sequential REST calls. Each fork took about 1 second. With 818 forks, that was 14 minutes. But GitHub Actions has a 6-minute timeout. The job was killed every time."

**The silent failure:** "Worse, the merge-upstream API returned 200 even when the sync did not happen. We had green checkmarks and zero actual syncs."

**v2:** "We switched to GraphQL for the fork list, `gh repo sync` for reliable syncing, and `asyncio.Semaphore(50)` for concurrency. Total time: 143 seconds. That is a 6x improvement. And we moved to Cloud Run so there is no timeout."

**The proof:** "Here is the SYNC_REPORT. 143s. Here is the LAST_RUN. 9 API calls, 4,876 rate limit remaining out of 5,000. We use 2.5% of our rate limit budget."

## Tailoring for Audiences

| Audience | Emphasize | Skip | Time |
|----------|-----------|------|------|
| Backend engineers | Decision log, CAP analysis, scale bottlenecks | Demo guide, high-level overview | 30 min |
| Frontend engineers | API endpoints, data shapes, search capabilities | Infrastructure decisions, CAP theorem | 15 min |
| Engineering managers | Tradeoffs, what is not working, timeline | Implementation details, Mermaid diagrams | 20 min |
| Product managers | Live stats, what users can do, what is missing | All technical details | 10 min |
| Interviewers (system design) | Decision log, CAP analysis, scale analysis, tradeoffs | Demo steps, working-with-engineers narrative | 45 min |
| Interviewers (behavioral) | Working with engineers, navigating ambiguity, audit story | Implementation details, diagrams | 30 min |

## Handling Breakage During a Demo

Things that might break and how to handle them:

**API is cold-starting (2-5 second delay):**
"This is on Neon's free tier. The database sleeps after inactivity. The first request wakes it up. In production, we would configure minimum instances."

**forksync had errors:**
"The SYNC_REPORT shows 1 error out of 792. That is a 99.87% success rate. The error is logged and we can investigate. The system is designed to tolerate individual fork failures."

**index.json shows fewer repos than /stats:**
"The raw file on GitHub may be cached for up to 24 hours. This is a known tradeoff (see doc 05, tradeoff 3). The API reads from Neon, which is always current."

**Ingestion data is empty:**
"Expected. Ingestion is not running yet. We prioritized sync reliability first. See doc 05, tradeoff 1."

The pattern: acknowledge the issue, explain why it happens, point to the documentation that anticipated it. Never be surprised by a known limitation.
